# v1.1.24 — Desktop storage: localStorage → filesystem IPC

## Problema
Il gioco usava `localStorage` per salvataggi e preferenze audio — un'API browser
non appropriata per un'app desktop. I dati finivano nel profilo Chromium, non in
una cartella dell'applicazione accessibile e portabile.

## Soluzione

### Nuovo layer IPC (Electron)
- `electron/preload.ts` (nuovo): espone `window.electronAPI` tramite `contextBridge`
  con 4 metodi: `writeSlot`, `readSlot`, `writeSettings`, `readSettings`
- `electron/main.ts`: aggiunto `preload: path.join(__dirname, 'preload.js')` a
  `webPreferences`; aggiunti 4 handler `ipcMain.handle` che leggono/scrivono:
  - Salvataggi: `<userData>/saves/slot-{id}.json` | `autosave.json`
  - Preferenze: `<userData>/settings.json`

### storageService.ts (nuovo)
Livello di astrazione unico per tutta la persistenza:
- Rileva automaticamente se `window.electronAPI` è disponibile (Electron)
  o usa localStorage come fallback (modalità dev web pura)
- Preferenze audio: caricate in memoria una volta sola (`initStorageSettings`)
  per permettere lettura sincrona da `audioService`
- Salvataggi: API async (`writeSlotData` / `readSlotData`)

### audioService.ts
- Sostituiti tutti gli accessi `localStorage.getItem/setItem` con
  `getSettingSync` / `setSetting` da `storageService`
- Rimosso il fallback `webkitAudioContext`: non necessario in Electron
  (Chromium moderno ha sempre `AudioContext`)

### SaveLoadOverlay.tsx
- `writeSlot`, `writeAutosave`, `loadSlotState` ora `async`
- Lettura slot all'apertura: `Promise.all` asincrono invece di accessi sincroni

### App.tsx
- `useEffect([], [])` all'avvio chiama `initStorageSettings()`
- Rimosso double-write `writeSlot` in `handleSlotSave` (il write era già
  fatto dall'overlay; App si occupava solo del feedback visivo)
