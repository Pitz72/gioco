## Problema risolto: GR4 — Commenti di debug/frustrazione visibili nel sorgente

### Analisi

In `game/rooms/ponteDiComando.ts`, il callback `onUse` dell'item `mappa_stellare` conteneva 9 righe di commenti che non spiegavano la logica del codice, ma documentavano il processo mentale del momento della scrittura:

```typescript
onUse: (state) => {
    // Hack per gestire "ESAMINA MAPPA" che triggera la conoscenza
    // Ma onUse è per USA.
    // Allora forse dovrei...
    // No aspetta, il problema è che...
    // Maledizione.
    // Allora devo usare un comando custom per ESAMINA MAPPA se voglio settare il flag.
    // Controllo types.ts per vedere se posso aggiungere un side-effect all'item examine...
    // No, non posso. Commands è la strada giusta.
    // Lasciamo questo onUse come fallback.
    return { description: "Non puoi usarla, è un ologramma.", eventType: 'error' };
}
```

Questi commenti:
- Non spiegano **cosa** fa il codice o **perché** è così strutturato
- Documentano un processo di debug che non ha valore per chi legge il sorgente in futuro
- Contengono il termine "Hack" che suggerisce codice non intenzionale (la soluzione attuale è invece corretta e intenzionale)
- Il parametro `state` era dichiarato ma non usato, causando un warning TypeScript

### Correzione

```typescript
// Prima
onUse: (state) => {
    // [9 righe di commenti debug/frustrazione]
    return { description: "Non puoi usarla, è un ologramma.", eventType: 'error' };
}

// Dopo
onUse: (_state) => {
    // USA MAPPA non ha senso: è un ologramma proiettato.
    // Il flag knowsAboutTrinarySystem viene settato dal comando ESAMINA MAPPA
    // definito nella lista commands, che permette il side-effect sullo stato.
    return { description: "Non puoi usarla, è un ologramma.", eventType: 'error' };
}
```

I due commenti rimasti spiegano concisamente:
1. Perché `onUse` restituisce un errore (semantica: un ologramma non si "usa")
2. Dove avviene il side-effect reale (flag `knowsAboutTrinarySystem`) e perché non è qui

Il parametro è stato rinominato `_state` per segnalare esplicitamente che è intenzionalmente non utilizzato.
