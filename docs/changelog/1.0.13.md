# Changelog v1.0.13 — Rimozione `dangerouslySetInnerHTML` dall'Output di Gioco

**Data:** 2026-03-20
**Tipo:** Fix criticità media (M7)
**Categoria:** Architettura / Sicurezza rendering

---

## Problema risolto: M7 — `dangerouslySetInnerHTML` usato con contenuto semi-dinamico

### Analisi

`TerminalOutput.tsx` renderizzava ogni riga dell'output di gioco tramite `dangerouslySetInnerHTML`:

```tsx
<div dangerouslySetInnerHTML={{ __html: line }}></div>
```

L'array `output: string[]` in `App.tsx` mescolava:
- **Testo puro**: descrizioni delle stanze, risposte ai comandi, messaggi di sistema — tutto proveniente da `processCommand()` e da stringhe costanti in `App.tsx`
- **HTML markup**: prompt di continuazione (`<span class="animate-blink">`) e testo con wrapper `<div class="whitespace-pre-wrap">` — generati direttamente in `App.tsx`

Il rischio dichiarato nell'analisi: se una descrizione di stanza avesse contenuto caratteri `<`, `>`, o `&` non escaped, avrebbe prodotto rendering HTML anomalo. Con `dangerouslySetInnerHTML` su *tutto* l'output, qualsiasi stringa venisse aggiunta all'array veniva interpretata come HTML, rendendo il sistema fragile per espansioni future.

### Correzione

**1. Nuovo tipo `OutputLine` in `types.ts`:**

```typescript
export type OutputLine =
    | { kind: 'text'; content: string }
    | { kind: 'html'; content: string };
```

Separa esplicitamente le righe che contengono markup UI (e che devono essere renderizzate come HTML) da quelle che contengono testo narrativo (che devono essere renderizzate come testo React puro).

**2. `TerminalOutput.tsx` aggiornato:**

```tsx
{output.map((line, index) =>
    line.kind === 'html'
        ? <div key={index} dangerouslySetInnerHTML={{ __html: line.content }} />
        : <div key={index}>{line.content}</div>
)}
```

`dangerouslySetInnerHTML` viene ora usato **solo** per le righe `kind: 'html'`, che sono generate esclusivamente da markup hardcoded in `App.tsx` (il prompt "PREMI INVIO PER CONTINUARE"). Il testo di gioco è sempre `kind: 'text'` e viene renderizzato come text node React — immune a qualsiasi injection.

**3. `App.tsx` aggiornato** — tutti i siti che pushano sull'array `output`:
- Testo narrativo (risposte stanza, comandi, messaggi sistema) → `{ kind: 'text', content: ... }`
- Prompt di continuazione (HTML con classe animate-blink) → `{ kind: 'html', content: ... }`

**Scope escluso:** `PaginatedScreen` (schermata istruzioni e intro) usa ancora `dangerouslySetInnerHTML` per il suo `displayedText`, ma il contenuto di `instructionsText` è HTML hardcoded puro — nessun contenuto dinamico o input utente — quindi rimane invariato.

### Risultato architetturale

- Il percorso critico (descrizioni stanze → output terminale) non passa più per `innerHTML`
- Il confine tra "contenuto sicuro" e "markup HTML" è ora esplicito nel tipo
- Future espansioni che aggiungono testo narrativo producono automaticamente `kind: 'text'` e non possono iniettare HTML inavvertitamente

### Verifica

- `npx tsc --noEmit` → zero errori ✅
- Nessuna dipendenza aggiunta ✅
- Comportamento visivo invariato ✅
