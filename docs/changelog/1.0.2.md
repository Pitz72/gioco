## 2. Migrazione Tailwind CSS 3 → 4

Tailwind CSS 4 introduce un cambio radicale nel sistema di configurazione.

### File eliminati
- **`tailwind.config.js`** — non esiste più in Tailwind 4. La configurazione avviene ora direttamente in CSS tramite `@theme {}`.
- **`postcss.config.js`** — non necessario con il plugin Vite `@tailwindcss/vite`. Eliminato.

### File modificati

**`index.css`** — aggiornate le direttive di importazione:
```css
/* Prima (Tailwind 3) */
@tailwind base;
@tailwind components;
@tailwind utilities;

/* Dopo (Tailwind 4) */
@import "tailwindcss";
```
Tutti gli stili custom (`scanline`, `animate-blink`, `no-scrollbar`) sono stati preservati invariati.

**`vite.config.ts`** — aggiunto plugin Tailwind 4:
```ts
import tailwindcss from '@tailwindcss/vite';
// ...
plugins: [tailwindcss(), react()]
```

**Vantaggio:** In Tailwind 4 la scansione dei file sorgente è automatica (nessun array `content:` da mantenere manualmente). I file in `game/rooms/` ora vengono scansionati correttamente senza configurazione aggiuntiva (risolve criticità **M6** dell'analisi).

## 4. Risoluzione Criticità di Sicurezza

### G3 — Rimosso importmap CDN esterno (GRAVISSIMA)
**File:** `index.html`

Rimosso il blocco `<script type="importmap">` che puntava a `https://aistudiocdn.com/` per React, React-DOM, lodash e @google/genai. Era un residuo dell'ambiente di sviluppo Google AI Studio, presente erroneamente anche nel build di produzione (`dist/index.html`). Costituiva un potenziale vettore di supply chain attack e rendeva l'app dipendente da un CDN esterno.

```html
<!-- RIMOSSO da index.html -->
<script type="importmap">
{
  "imports": {
    "react": "https://aistudiocdn.com/react@^19.2.0",
    ...
  }
}
</script>
```

Il build Vite bundla già tutte le dipendenze in autonomia: l'importmap era completamente ridondante e pericoloso.

### GR3 — Rimossi `@google/genai` e `geminiService.ts` (GRAVE)
**File:** `package.json`, `services/geminiService.ts`

La libreria `@google/genai ^1.28.0` era listata come dipendenza di **produzione** e veniva bundlata nell'output finale, gonfiando inutilmente il bundle. Il file `services/geminiService.ts` era completamente vuoto (una riga vuota). Entrambi eliminati.

### M5 — Rimossi `define` per variabili API da vite.config.ts (MEDIA)
**File:** `vite.config.ts`

Rimosse le righe:
```ts
'process.env.API_KEY': JSON.stringify(env.GEMINI_API_KEY),
'process.env.GEMINI_API_KEY': JSON.stringify(env.GEMINI_API_KEY)
```
Se fosse stato aggiunto un file `.env` con una chiave API reale, Vite l'avrebbe inlineata in chiaro nel bundle distribuito. Rischio eliminato alla radice.

## 6. Risoluzione Criticità M6 — Tailwind non scannerizzava `game/rooms/`

Con Tailwind 3 il file `tailwind.config.js` aveva:
```js
content: ["./index.html", "./App.tsx", "./components/**/*.{js,ts,jsx,tsx}"]
```
I file di stanza (`game/rooms/*.ts`, `game/*.ts`) non erano inclusi. Se avessero contenuto classi Tailwind, sarebbero state purgat in produzione.

Con Tailwind 4 e `@tailwindcss/vite`, la scansione è automatica su tutti i file del progetto (esclusi quelli in `.gitignore`). Criticità risolta strutturalmente, senza configurazione manuale.
